# 合同全量要素系统设计文档

# 1. 系统概述

## 1.1 系统定位

本系统是一款专注于合同全量要素管理的中间件系统，核心定位为连接业界主流SaaS CRM系统与企业内部业务系统的合同数据枢纽。系统通过集成各CRM系统的合同数据，构建统一的合同全量要素数据模型，支持字段扩展、AI与人工补充完善，并通过规则引擎与AI引擎实现合同数据问题的智能识别与预警，同时提供灵活的多租户权限管控与字段展示差异化配置能力，最终实现合同数据的标准化、精细化管理，以及便捷的前端集成与多场景复用。

## 1.2 核心目标

- 实现与主流SaaS CRM系统的高效对接，保障合同数据同步的实时性与准确性。

- 构建可扩展的合同字段体系，支持基于基础CRM数据的人工与AI驱动的字段补充扩展。

- 通过规则引擎+AI引擎实现合同数据问题的自动识别、提示与多渠道通知确认。

- 提供结构化、可视化的合同数据展示能力，支持嵌入源CRM系统的前端卡片。

- 支持逻辑多租户架构，实现租户级别的功能权限与数据维度权限隔离。

- 提供面向多租户的服务返回字段与前端展示字段差异化配置能力。

## 1.3 适用范围

本系统适用于需要整合多SaaS CRM合同数据、进行合同精细化管理、要求合同风险前置识别的企业，尤其适合多业务线、多分支机构、需要进行租户隔离管理的集团型企业或SaaS服务提供商。

# 2. 核心需求分析

## 2.1 数据集成与实时同步需求

- 支持对接Salesforce、HubSpot、钉钉宜搭、企业微信CRM等主流SaaS CRM系统。

- 实现CRM系统合同数据的全量同步与增量实时同步，同步延迟不超过3秒。

- 支持同步数据的清洗、转换与标准化处理，适配系统统一数据模型。

## 2.2 合同字段扩展需求

- 支持基于CRM同步的基础合同字段，扩展自定义字段（文本、数字、日期、枚举、附件等类型）。

- 支持人工手动补充完善扩展字段数据，支持AI自动提取合同文本（PDF、Word等格式）中的关键信息补充扩展字段。

- 扩展字段需与基础字段关联，支持按合同维度统一查询与管理。

## 2.3 合同数据问题识别与通知需求

- 支持通过可视化规则引擎配置合同数据校验规则（如字段缺失、格式错误、逻辑冲突、风险阈值超标等）。

- 支持AI引擎辅助识别复杂合同数据问题（如条款歧义、潜在风险条款、数据与合同文本不一致等）。

- 问题识别后，支持实时生成预警提示，并通过系统内消息、邮件、短信以及源CRM系统回调通知等方式推送至相关责任人。

- 支持问题的认领、处理、复核与闭环管理，并记录全流程日志。

## 2.4 数据展示与前端集成需求

- 提供结构化的合同全量数据展示界面，支持列表查询、详情查看、筛选、排序、导出等功能。

- 支持开发轻量级前端卡片组件，可嵌入源CRM系统前端页面，实现合同关键信息与问题预警的快速展示。

- 前端展示支持响应式设计，适配不同终端设备。

## 2.5 多租户与权限管控需求

- 支持逻辑多租户架构，各租户数据相互隔离，共享系统核心服务与资源。

- 支持租户级别的功能权限控制（如是否开启AI字段补充、是否可配置规则引擎等）。

- 支持租户内数据维度权限控制（如按部门、业务线、区域等维度划分数据访问权限）。

## 2.6 字段差异化配置需求

- 支持多租户自定义配置服务接口返回字段（如部分字段隐藏、字段别名映射等）。

- 支持多租户自定义配置前端展示字段（如展示顺序、是否显示、字段标签、必填标识等）。

- 配置变更实时生效，无需重启系统。

# 3. 系统架构设计

## 3.1 架构总体设计

本系统采用分层微服务架构，基于云原生技术栈构建，确保系统的高可用性、可扩展性与可维护性。整体架构从上至下分为前端层、接入层、核心服务层、数据层、基础设施层，同时引入多租户治理中心与权限中心实现全局管控。

## 3.2 各层架构设计

### 3.2.1 前端层

- 租户管理后台：基于Vue3+Element Plus开发，提供租户配置、权限管理、合同数据管理、规则配置等功能模块。

- 嵌入型前端卡片：基于Web Component开发，支持嵌入主流CRM系统前端，提供轻量化合同信息展示与交互功能。

- 公共组件库：封装表格、表单、弹窗、消息提示等通用组件，保障前端开发效率与交互一致性。

### 3.2.2 接入层

- API网关：基于Spring Cloud Gateway实现，负责请求路由、负载均衡、限流、鉴权、租户上下文解析等功能。

- WebHook接收服务：专门接收各CRM系统的WebHook推送消息，实现合同数据的增量实时同步。

- 接口适配层：提供标准化的API接口，同时支持对接不同CRM系统的适配器开发，实现多CRM系统的统一接入。

### 3.2.3 核心服务层

- 租户管理服务：负责租户的注册、审核、信息管理、租户状态管控等功能。

- 数据集成服务：负责与各CRM系统的全量/增量数据同步，实现数据清洗、转换、标准化与入库。

- 合同字段管理服务：负责基础字段与扩展字段的定义、维护、字段关联等功能。

- AI辅助服务：负责合同文本解析、关键信息提取、复杂问题识别等AI能力的封装与调用。

- 规则引擎服务：提供可视化规则配置、规则解析、规则执行与结果返回等功能，支持与AI引擎协同工作。

- 问题管理服务：负责合同数据问题的识别、记录、预警、处理流程管理与日志追溯。

- 通知服务：负责整合多种通知渠道（系统消息、邮件、短信、CRM回调等），实现问题通知的精准推送。

- 权限管理服务：基于RBAC模型，实现租户级别的功能权限与数据维度权限的定义、分配与校验。

- 配置管理服务：负责多租户的服务返回字段与前端展示字段的差异化配置管理。

### 3.2.4 数据层

- 关系型数据库：采用PostgreSQL（PG）数据库（主从架构），存储租户信息、权限数据、字段配置、规则配置、问题记录等结构化数据。

- 缓存：采用Redis Cluster，缓存热点合同数据、租户配置、权限信息等，提升系统响应速度。

- 对象存储：采用MinIO，存储合同附件、AI解析结果文件等非结构化数据。

- 消息队列：采用Kafka，实现服务间异步通信（如数据同步通知、问题预警推送等），解耦服务依赖。

## 3.3 多租户隔离设计

系统采用逻辑多租户隔离方案，通过以下方式保障租户数据与配置的隔离性：

- 数据隔离：所有业务数据表均包含租户ID字段，通过租户ID进行数据过滤；核心配置表按租户维度存储配置信息，查询时携带租户ID上下文。

- 服务隔离：通过租户上下文传递机制，确保所有服务调用均携带租户标识，实现服务逻辑的租户差异化处理。

# 4. 核心模块详细设计

## 4.1 数据集成与实时同步模块

### 4.1.1 模块功能

实现与主流SaaS CRM系统的对接，完成合同数据的全量同步、增量实时同步，以及同步数据的清洗、转换与标准化处理。

### 4.1.2 同步方案设计

- 全量同步：采用定时任务触发（可配置同步周期），通过调用各CRM系统的批量查询接口获取全量合同数据，同步完成后生成全量同步报告。

- 增量实时同步：优先采用CRM系统的WebHook机制，当CRM合同数据发生变更时，主动推送变更数据至本系统的WebHook接收服务；对于不支持WebHook的CRM系统，采用CDC（变更数据捕获）技术，通过监听CRM数据库的binlog日志获取增量变更数据，确保同步实时性。

- 数据清洗与转换：基于租户配置的字段映射规则，对同步数据进行格式校验、数据类型转换、缺失值处理、重复数据去重等操作，将异构数据转换为系统统一的合同数据模型。

### 4.1.3 关键流程

1. 租户配置CRM对接信息（API密钥、访问地址、WebHook回调地址等）。

2. 系统初始化全量同步任务，调用CRM批量查询接口获取全量合同数据。

3. 对全量数据进行清洗、转换，入库至合同基础数据表，并记录同步日志。

4. 启用增量同步机制，通过WebHook或CDC监听CRM数据变更。

5. 接收增量变更数据，进行清洗转换后更新合同基础数据表，同时触发后续的字段补充、问题识别等流程。

6. 同步失败时，记录失败原因，触发重试机制（重试3次，间隔10秒），重试失败后发送告警通知。

## 4.2 合同字段扩展模块

### 4.2.1 模块功能

支持基础字段的管理、扩展字段的定义与维护，以及通过人工或AI方式补充扩展字段数据。

### 4.2.2 字段体系设计

- 基础字段：由CRM同步数据映射生成，包括合同编号、客户名称、合同金额、签约日期、合同状态等核心字段，字段类型与长度固定。

- 扩展字段：租户自定义添加，支持文本、数字、日期、枚举、单选、多选、附件等多种类型，可配置字段名称、字段标识、是否必填、默认值、数据校验规则等属性。

- 字段关联：扩展字段通过合同ID与基础字段关联，形成完整的合同全量要素数据模型。

### 4.2.3 字段补充方式

- 人工补充：租户用户通过系统管理后台的合同详情页面，手动填写或上传附件补充扩展字段数据。

- AI自动补充：用户上传合同文本文件（PDF、Word等），AI辅助服务通过OCR识别、NLP文本解析技术，提取合同中的关键信息（如付款方式、违约责任、有效期等），自动填充至对应的扩展字段，用户可对AI提取结果进行审核与修正。

## 4.3 规则引擎+AI引擎模块

### 4.3.1 模块功能

实现合同数据问题的规则化识别与AI辅助识别，生成问题预警并触发通知流程。

### 4.3.2 规则引擎设计

- 规则配置：提供可视化规则配置界面，支持租户用户通过拖拽、选择等方式配置规则条件（如“合同金额>100万且未填写付款担保方式”“签约日期早于合同生效日期”等）、规则级别（提示、警告、严重）、触发时机（数据同步后、字段更新后、定时巡检）与执行动作（生成预警、发送通知）。

- 规则引擎核心：基于drools规则引擎实现，负责规则的解析、编译与执行，支持规则的批量执行与优先级排序。

- 规则管理：支持规则的新增、修改、删除、启用/禁用、版本管理等功能，确保规则变更的可追溯性。

### 4.3.3 AI引擎设计

- 核心能力：集成OCR识别、NLP文本分析、合同风险识别等AI模型，实现合同文本与结构化数据的一致性校验、复杂条款歧义识别、潜在风险条款（如不公平格式条款、模糊责任条款）识别等功能。

- 模型迭代：支持模型的持续训练与优化，租户用户可将人工审核修正后的结果反馈至AI模型，提升模型识别准确率。

- 调用方式：AI引擎以服务接口形式提供，规则引擎可根据规则配置调用AI引擎能力，实现复杂问题的辅助识别。

### 4.3.4 问题识别与处理流程

1. 触发时机：数据同步完成后、扩展字段补充更新后，或定时巡检任务触发。

2. 问题识别：规则引擎执行已配置的规则，同时调用AI引擎辅助识别复杂问题，汇总识别结果。

3. 问题分级：根据规则级别或AI识别的风险等级，将问题分为提示（不影响合同效力，需关注）、警告（可能存在风险，需核实）、严重（存在重大风险，需立即处理）三个级别。

4. 预警生成：系统记录问题详情（问题描述、关联合同ID、识别时间、识别方式等），生成问题预警记录。

5. 通知推送：通知服务根据问题级别与配置的通知方式，将预警信息推送至相关责任人（如合同负责人、风控人员）。

6. 问题处理：责任人接收通知后，对问题进行认领、核实、处理（如补充数据、修正错误、确认风险可接受），处理完成后提交审核，审核通过则问题闭环，未通过则返回重新处理。

## 4.4 多租户权限管理模块

### 4.4.1 模块功能

实现租户的功能权限与数据维度权限的定义、分配、校验与管理。

### 4.4.2 权限模型设计

采用RBAC（角色基础访问控制）模型，并扩展数据维度权限控制：

- 功能权限：定义系统功能模块的操作权限（如“查看合同数据”“配置规则引擎”“扩展字段管理”等），通过角色关联权限，租户用户关联角色，实现功能权限的批量分配。

- 数据维度权限：支持按部门、业务线、区域、合同类型等维度划分数据访问范围，租户管理员可为角色配置数据维度权限，确保用户仅能访问其权限范围内的合同数据。

### 4.4.3 权限校验流程

1. 用户发起请求时，API网关解析租户ID与用户信息，传递至权限管理服务。

2. 权限管理服务查询用户关联的角色，以及角色对应的功能权限与数据维度权限。

3. 校验用户是否具备当前请求的功能权限，若不具备则返回权限不足错误。

4. 若具备功能权限，进一步校验用户的数据维度权限，在查询数据时添加数据维度过滤条件，确保返回数据符合权限要求。

## 4.5 字段差异化配置模块

### 4.5.1 模块功能

支持多租户自定义配置服务接口返回字段与前端展示字段，实现差异化的字段展示与数据返回。

### 4.5.2 配置设计

- 服务返回字段配置：租户管理员可选择系统中的基础字段与扩展字段，配置哪些字段需要在API接口返回中展示，支持设置字段别名、字段排序顺序，可隐藏敏感字段。

- 前端展示字段配置：租户管理员可配置前端页面（管理后台、嵌入卡片）中展示的字段，支持设置字段展示名称、展示顺序、是否显示、是否可编辑、字段样式（如颜色、字体）等属性。

- 配置存储：配置信息按租户维度存储在配置表中，支持配置的导出、导入与版本管理。

### 4.5.3 配置生效流程

1. 租户管理员在系统管理后台完成字段差异化配置，并提交生效。

2. 配置管理服务将配置信息更新至数据库，并同步缓存至Redis。

3. 当用户调用API接口时，接口服务从缓存中获取当前租户的返回字段配置，动态组装返回数据。

4. 当前端页面加载时，前端组件从缓存中获取当前租户的展示字段配置，动态渲染页面组件。



## 4.6 扩展字段数据修改功能

支持租户用户对已补充的扩展字段数据进行修改操作，满足数据纠错、信息更新等场景需求，具体能力如下：

- 修改权限控制：基于租户内角色权限体系，仅授权用户可对扩展字段数据进行修改，确保数据修改的安全性与规范性。

- 修改场景适配：支持人工补充字段的直接编辑修改，以及AI补充字段的二次修正，修改后保留修改痕迹，便于追溯。

- 关联影响同步：扩展字段数据修改后，自动同步至依赖该字段的查询结果、统计报表及问题识别规则，确保数据一致性。

## 4.7 全量数据修改审计功能

为保障数据修改的可追溯性与合规性，系统实现全量数据修改审计功能，覆盖基础字段与扩展字段的所有修改操作，具体设计如下：

- 审计日志记录：每次数据修改后，自动记录审计日志，包含修改人ID/姓名、修改时间、修改字段名称、原始值、修改后值、修改方式（人工/AI/系统同步）、修改原因（可选，支持用户填写）等关键信息。

- 审计日志管理：审计日志按租户维度隔离存储，支持租户管理员按时间范围、字段类型、修改人等条件查询审计日志，支持日志导出功能，满足合规检查需求。

- 修改追溯能力：通过审计日志可完整追溯任意字段的修改历史，明确各版本数据的修改主体与时间节点，为数据争议处理、合规审计提供依据。

# 5. 数据设计

## 5.1 核心数据模型

### 5.1.1 租户信息表（tenant_info）

|字段名|字段类型|是否主键|字段说明|
|---|---|---|---|
|tenant_id|varchar(64)|是|租户唯一标识|
|tenant_name|varchar(128)|否|租户名称|
|contact_person|varchar(64)|否|联系人|
|contact_phone|varchar(20)|否|联系电话|
|status|tinyint(1)|否|租户状态（0-未激活，1-正常，2-禁用）|
|create_time|datetime|否|创建时间|
|update_time|datetime|否|更新时间|
### 5.1.2 合同基础信息表（contract_base）

|字段名|字段类型|是否主键|字段说明|
|---|---|---|---|
|contract_id|varchar(64)|是|合同唯一标识（系统内主键）|
